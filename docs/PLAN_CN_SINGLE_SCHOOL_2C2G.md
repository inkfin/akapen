# 中国高中作文批改实施计划（单校年级 / 2C2G / 国产 API）

## 1. 目标与约束

- 场景：单个学校、单个年级的作文批改，小批量多批次运行。
- 资源：部署机仅 `2 CPU / 2 GB RAM`，不承担本地大模型推理。
- 要求：可调用国产模型 API，自主实现与自主部署，支持一键批处理与断点续跑。
- 结果：输出可导入现有校内系统的成绩与评语，保留复核与审计依据。

## 2. 总体方案

采用 "轻量本地编排 + 国产云端模型推理" 架构：

1. 本地服务负责任务编排、文件管理、规则校验、质量闸门、结果导出。
2. 云端通过国产视觉模型 API 完成识别与评分（单轮、结构化输出）。
3. 本地不做重型中间件，使用 SQLite + 本地文件目录，保证低资源可运行。

## 3. 技术选型（MVP）

- API 框架：`FastAPI`
- 任务执行：`asyncio + semaphore`
- 存储：`SQLite`（WAL 模式）
- 文件：本地磁盘目录（输入/处理中间件/输出）
- 图像处理：`Pillow`（旋转矫正、压缩、尺寸统一）
- 导出：`CSV + JSON`
- 日志：滚动日志（按天或按大小切分）

## 4. 目录与数据规范

建议目录：

```text
data/
  input/        # 原始作文图片或PDF
  work/         # 预处理后的单页图
  output/       # 评分结果、统计报表、复核清单
  archive/      # 历史批次归档
```

建议命名：

- 学生主键：`school_year + class_id + student_id`
- 作文主键：`exam_id + prompt_id + student_id`
- 任务主键：`batch_id + student_id + attempt`

## 5. 批改流程（单次批处理）

1. 导入名单和作业（CSV + 文件夹映射）。
2. 预处理图片（方向、分辨率、压缩）。
3. 组装评分输入（题目、规则、学生图片、元信息）。
4. 调用国产 VL API（单轮，返回严格 JSON）。
5. 规则引擎校验（总分边界、扣分项完整性、字段完整性）。
6. 质量闸门分流（自动通过 vs 人工复核）。
7. 导出结果（成绩 CSV、评语 CSV、复核清单 CSV、原始 JSON）。
8. 写入批次报告（耗时、失败率、复核率、异常统计）。

## 6. 模型调用规范（关键）

每份作文默认只调用一次主模型，失败时最多重试一次：

- `timeout_sec`: 60~90
- `max_attempts`: 2
- `temperature`: 0~0.2（稳定评分）
- 输出格式：强制 JSON schema

建议输出字段：

- `final_score`（总分）
- `dimension_scores`（分项分）
- `deductions`（扣分点和依据）
- `feedback`（评语）
- `confidence`（0~1）
- `evidence`（原文证据片段）

## 7. 2C2G 参数建议（默认值）

- 并发数：`6`（可在 `4~10` 之间调优）
- 批次提交粒度：每批 `30~50` 份
- 落盘频率：每 `10` 份即时写库+写文件
- 最大内存占用策略：逐份读取、逐份释放，不保留大图缓存

调优顺序：

1. 先调超时（避免假死）。
2. 再调并发（避免 API 429）。
3. 最后调批次大小（平衡吞吐与可恢复性）。

## 8. 评分规则模板（先标准化）

每次任务绑定一份规则文件（版本化）：

- `rubric_id`
- `rubric_version`
- `total_score`
- `dimensions`
- `deduction_rules`
- `must_fail_conditions`
- `feedback_style`

要求：

- 同一批次固定同一规则版本，不允许中途漂移。
- 每个扣分必须有证据。
- 所有结果可追溯到规则项。

## 9. 质量闸门与人工复核

自动进入复核池的条件（建议）：

- `confidence < 0.65`
- 分数边界（如 59/60, 89/90）
- 缺失证据字段或 JSON 不完整
- 评语情绪与分数冲突
- 书写质量差导致识别不稳定

复核策略：

- 优先复核边界分和低置信度样本。
- 复核后写回最终分，并记录修订原因。

## 10. 对接现有校内系统

优先级从高到低：

1. CSV 批量导入导出（最快落地）
2. HTTP API 对接（中期最佳）
3. RPA 自动回填（无 API 时兜底）

最小对接字段：

- `student_id`
- `name`
- `class_id`
- `score`
- `feedback`
- `review_flag`

## 11. 安全与合规（学校场景）

- 学生身份信息与作文内容最小化传输。
- API Key 使用环境变量，不写入代码库。
- 导出文件脱敏（公开统计不含姓名学号）。
- 保留操作日志和评分快照，支持校内抽查。

## 12. 4 天落地排期（可执行）

### Day 1：骨架与数据流

- 建立目录结构、SQLite 表、批次状态机。
- 打通 "导入名单 -> 匹配作业 -> 建任务"。

### Day 2：模型接入与单份跑通

- 接入 1 个国产 VL API Provider。
- 固定 prompt 与 JSON schema，完成单份评分闭环。

### Day 3：批处理与质量闸门

- 加入并发、重试、断点续跑。
- 实现复核池判定与复核导出清单。

### Day 4：小规模验收

- 用 100~300 份历史样本试跑。
- 调整阈值，形成标准作业流程（SOP）。

## 13. 验收标准（首版）

- 批次完成率 `>= 98%`
- 自动通过率 `>= 70%`（其余进入复核）
- 复核后与教研组一致率 `>= 90%`
- 单份平均耗时、单批总耗时、失败重试率可统计

## 14. 上线后迭代清单

- 增加第二模型作为回退，降低供应商单点风险。
- 按年级/题型维护多套 rubric 模板。
- 累积复核样本，优化阈值与提示词。
- 增加批次对比报表（班级、题型、分项能力趋势）。

## 15. 本项目下一步实现建议（与当前仓库结合）

1. 在 `core/providers/` 增加目标国产模型 Provider（若已有可复用则配置化）。
2. 在 `core/grader.py` 增加严格 JSON 输出与校验失败重试。
3. 在 `core/storage.py` 增加 `confidence/evidence/review_flag` 字段落盘。
4. 在 UI 增加 "质量闸门阈值" 与 "复核清单导出"。
5. 在 `data/exports/` 增加标准化 CSV 导出模板（便于校内系统导入）。

## 16. 题目上下文支持（schema v2，2026-05 加入）

为了让前端（`web/`）能在「按题批改」时把题干 / 评分要点送进来，
`TaskCreateRequestJSON` / multipart Form 新增可选字段 `question_context`（≤4000 字）。
`grading_tasks` 表用 `_apply_v2()` 增列；`core.grader._prepend_question_context`
把 ctx 拼到 prompt 顶部（不改 `prompts/grading.md` / `prompts/single_shot.md`，
现网部署热升级时无需重置 settings）。老用户不传该字段 → 行为完全等价于 v1。
